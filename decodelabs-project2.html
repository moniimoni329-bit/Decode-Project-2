Mission Brief

THE SCENARIO

Context

A startup is launching a new dynamic application and needs a dedicated server environment with full control over the Operating System to install custom software and security patches. As the assigned Cloud Infrastructure Engineer at DecodeLabs, the task is to provision, secure, and deploy a live web server entirely from scratch using cloud-native IaaS tools.

  • 🖥️Launch a Virtual Machine (EC2) running Amazon Linux 2023 on AWS.
  • 🔐Connect to the server securely via SSH key-pair authentication (.pem file).
  • 🌐Install and configure Nginx web server through the Linux command line.
  • 📄Deploy a custom "Welcome to DecodeLabs" HTML page publicly accessible via the server's IP.

Execution Log

THE 6 PHASES

01
PROVISIONING THE INSTANCE

Logged into the AWS Management Console → EC2 Dashboard → Launch Instance. Configured the following parameters:

Instance Config

  • Name: Server-Commander-01
  • AMI: Amazon Linux 2023
  • Type: t2.micro (Free Tier)
  • Key Pair: Created & Downloaded .pem

Why These Choices

  • Amazon Linux 2023 = AWS-optimized
  • t2.micro = Free Tier eligible
  • .pem key = secure SSH access
  • IaaS model = full OS control
02
SECURING THE PERIMETER — SECURITY GROUPS

Configured inbound firewall rules. Security Groups act as "allow-only" virtual firewalls — all traffic is denied by default.

TypePortSourceStatus
SSH22My IP Only 🔒 RESTRICTED
HTTP80Anywhere (0.0.0.0/0) 🌐 OPEN
HTTPS443Anywhere (0.0.0.0/0) 🌐 OPEN
03
THE HANDSHAKE — SSH CONNECTION

Secured the private key and established an SSH session from the local terminal to the EC2 instance.

# Step 1: Lock down the key file (mandatory on Mac/Linux) chmod 400 keyname.pem # Step 2: Connect to the EC2 instance via SSH ssh -i keyname.pem ec2-user@<Public-IPv4> # Output: Access Granted ✓ [ec2-user@ip-172-31-0-1 ~]$
04
DEPLOYING THE PAYLOAD — NGINX INSTALLATION

Nginx was selected over Apache for its event-driven, high-concurrency architecture. Three commands to bring the server online:

# Refresh system package repository sudo dnf update -y # Install Nginx web server daemon sudo dnf install nginx -y # Start and enable Nginx service sudo systemctl start nginx sudo systemctl enable nginx # Verify Nginx is running sudo systemctl status nginx ● nginx.service - Active: active (running) ✓
05
CUSTOMIZING THE ASSET — HTML PAGE DEPLOYMENT

Edited the Nginx default index file using the nano terminal editor to deploy the custom DecodeLabs welcome page.

# Open the web root index file for editing sudo nano /usr/share/nginx/html/index.html # Custom HTML content deployed: <html> <head> <title>DecodeLabs | Server Commander</title> </head> <body> <h1>Welcome to DecodeLabs: Mission Accomplished</h1> </body> </html> # Save: Ctrl+O → Enter Exit: Ctrl+X
06
VERIFICATION — SERVER IS LIVE

Opened a browser and navigated to the EC2 instance's public IP. The DecodeLabs page loaded successfully — server confirmed online.

Browser URL: http://<EC2-Public-IP> ● ONLINE Welcome to DecodeLabs: Mission Accomplished

What Was Learned

KEY CONCEPTS MASTERED

IaaS Model

AWS provides the physical hardware. The engineer owns the OS, software stack, and security configuration.

Hypervisor Logic

The AWS Nitro System partitions physical hardware into isolated Virtual Machines, maximizing workload resources.

SSH Key Auth

Public key stored on the server; private key (.pem) stored locally and never shared. Lose the key, lose the server.

Security Groups

Stateful allow-only firewalls. SSH restricted to one IP; web ports open to the world. Default: deny all.

Nginx vs Apache

Nginx: event-driven, high-concurrency, low memory. Apache: process-driven, flexible. Nginx selected for this mission.

Pets vs Cattle

Old way: nurse servers to health (pets). Cloud way: terminate and replace identical instances (cattle).
